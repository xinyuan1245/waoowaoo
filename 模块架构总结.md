# 模块架构总结

## 1. 项目定位

`waoowaoo` 是一套面向 AI 短剧 / 漫画视频生产的全流程系统。核心目标是把“文本理解 -> 资产生成 -> 分镜 -> 视频 -> 配音 -> 口型同步”串成一个可配置、可追踪、可复用的工作流。

整体不是单点 AI 调用，而是由以下几层组成：

- Web 应用层：Next.js 页面、API Route、前端工作台
- 业务域层：项目、资产、分镜、语音、视频等业务模型
- AI 运行层：模型配置、生成器、任务队列、Worker
- 媒体层：图片/音频/视频对象、存储、签名访问
- 支撑层：认证、计费、日志、测试、守卫脚本

## 2. 顶层模块划分

### 2.1 前端页面与工作区

主要目录：

- `src/app/[locale]/*`
- `src/components/*`
- `src/features/video-editor/*`

职责：

- 提供首页、登录、个人中心、工作区、资产中心
- 提供小说推广工作台的各阶段 UI
- 承载视频编辑器、分镜面板、资产管理面板
- 调用后端 API，并通过 SSE / React Query 同步任务状态

关键页面：

- `src/app/[locale]/workspace/page.tsx`
- `src/app/[locale]/workspace/asset-hub/page.tsx`
- `src/app/[locale]/workspace/[projectId]/*`
- `src/app/[locale]/profile/page.tsx`

### 2.2 API 层

主要目录：

- `src/app/api/*`

职责：

- 作为前后端边界
- 承接 UI 请求，做鉴权、参数校验、服务层编排
- 触发任务提交、读取项目数据、管理资产、管理配置

典型 API：

### 2.3 业务服务层

- `src/lib/assets/services/*`

- 承载核心业务规则
- 将 Prisma、AI 任务、媒体、计费等能力组装成稳定接口
- 避免 API Route 直接堆业务逻辑

### 2.4 AI 运行层

主要目录：

- `src/lib/model-gateway/*`

## 3. 核心业务域

### 3.1 项目域

核心模型：

- `Project`
- `NovelPromotionProject`
- `NovelPromotionEpisode`
- `NovelPromotionClip`
- `NovelPromotionStoryboard`
- `NovelPromotionPanel`
- `NovelPromotionVoiceLine`

定义位置：

- `prisma/schema.prisma`

职责：

- 维护一个创作项目的全生命周期数据
- 保存模型配置、文本、章节、分镜、面板、音频、视频等信息

特点：

- `Project` 更偏外层项目容器
- `NovelPromotionProject` 是小说推广工作流的主体数据
- 各阶段数据都围绕该项目组织

### 3.2 资产域

核心类型：

- 角色 `character`
- 场景 `location`
- 道具 `prop`
- 音色 `voice`

统一契约：

- `src/lib/assets/contracts.ts`

关键设计：

- 同时支持 `global` 和 `project` 两种 scope
- 资产有统一的前端摘要结构 `AssetSummary`
- 角色和场景类资产都支持“变体/候选 render/选中 render/上次 render”

#### 全局资产

主要表：

- `GlobalAssetFolder`
- `GlobalCharacter`
- `GlobalCharacterAppearance`
- `GlobalLocation`
- `GlobalLocationImage`
- `GlobalVoice`

作用：

- 作为跨项目复用的资产中心
- 可复制到项目内继续演化

#### 项目资产

主要表：

- `NovelPromotionCharacter`
- `CharacterAppearance`
- `NovelPromotionLocation`
- `LocationImage`

作用：

- 表示某个项目内部真实使用的角色/场景/道具

#### 资产读取

入口：

- `src/lib/assets/services/read-assets.ts`
- `src/app/api/assets/route.ts`

流程：

1. 按 scope 查询 Prisma
2. 附着媒体字段
3. 映射成统一 `AssetSummary`
4. 前端消费统一结构

#### 资产动作

入口：

- `src/lib/assets/services/asset-actions.ts`

支持：

- 创建资产
- 生成 render
- 修改 render
- 选中 render
- 回滚 render
- 复制全局资产到项目
- 更新资产元信息
- 更新变体信息

### 3.3 媒体域

核心表：

- `MediaObject`

核心文件：

- `src/lib/media/service.ts`
- `src/lib/media/attach.ts`

职责：

- 统一管理图片/音频/视频文件
- 通过 `storageKey`、`publicId` 建立稳定引用
- 为业务对象补充 `media` 引用和标准访问 URL

关键能力：

- `ensureMediaObjectFromStorageKey()`
- `resolveMediaRef()`
- `resolveStorageKeyFromMediaValue()`
- `attachMediaFieldsToProject()`
- `attachMediaFieldsToGlobalCharacter()` 等

设计意图：

- 业务表里保留旧 URL 字段，兼容历史数据
- 同时逐步迁移到 `mediaId -> MediaObject -> /m/publicId`

### 3.4 模型配置域

核心文件：

- `src/lib/api-config.ts`
- `src/lib/config-service.ts`
- `src/lib/model-config-contract.ts`
- `src/app/api/user/api-config/route.ts`

职责：

- 管理 provider、model、capabilities、默认模型
- 统一模型 key 规范：`provider::modelId`
- 支持官方 provider 和兼容协议 provider

支持的模型类型：

- `llm`
- `image`
- `video`
- `audio`
- `lipsync`

特点：

- 用户可以自定义模型
- 系统也有内置 preset
- 项目和用户偏好可以分别配置默认模型

## 4. AI 调用与执行架构

### 4.1 模型接入层

主要目录：

- `src/lib/providers/*`
- `src/lib/model-gateway/*`
- `src/lib/llm/*`

职责：

- 封装具体供应商 API
- 统一 LLM、图像、视频、音频调用
- 兼容 OpenAI 风格和 Gemini 风格协议

示例：

- `providers/bailian/*`
- `providers/siliconflow/*`
- `llm/providers/google.ts`
- `llm/providers/openai-compat.ts`
- `model-gateway/openai-compat/*`

### 4.2 生成器层

主要目录：

- `src/lib/generators/*`

职责：

- 面向业务的图像、视频、音频生成入口
- 对 provider 差异做更高一层抽象

示例：

- `generators/image/*`
- `generators/video/*`
- `generators/audio/*`

### 4.3 任务系统

主要目录：

- `src/lib/task/*`
- `src/lib/workers/*`
- `src/lib/run-runtime/*`
- `scripts/watchdog.ts`
- `scripts/bull-board.ts`

职责：

- AI 任务不直接同步执行，而是进入队列
- Worker 异步消费任务
- 前端通过任务状态和 SSE 订阅结果

典型能力：

- 提交任务
- 去重
- 心跳与超时
- 重试
- 状态流转
- 事件发布

关键说明：

- 图片生成、图片修改、文本分析、语音生成、视频生成都走任务系统
- Bull Board 用于查看队列情况

### 4.4 工作流编排

主要目录：

- `src/lib/workers/handlers/*`
- `src/lib/novel-promotion/*`
- `src/lib/storyboard-phases.ts`

职责：

- 把单个 AI 能力串成完整业务阶段
- 例如小说分析、角色生成、场景生成、分镜构建、视频阶段等

## 6. 典型数据流

### 6.1 资产读取流

1. 前端调用 `/api/assets`
2. API 校验 scope / 鉴权
3. `readAssets()` 从 Prisma 查询
4. `attachMediaFields...()` 补媒体引用
5. `map...ToAsset()` 转成统一 `AssetSummary`
6. 前端渲染

### 6.2 资产生成流

1. 前端点击“生成图片”
2. API 调 `submitAssetGenerateTask()`
3. 服务层读取项目或用户模型配置
4. 构建计费 payload 和任务 payload
5. 任务进入队列
6. Worker 调用具体生成器
7. 结果写回资产表和媒体表
8. SSE / 轮询更新前端状态

### 6.3 复制全局资产到项目

1. 前端选择全局资产
2. API 调 `copyAssetFromGlobal()`
3. 服务层读取全局资产
4. 重新创建项目侧资产记录
5. 记录 `sourceGlobal...Id`
6. 项目内继续独立使用

## 8. 当前架构特点

### 优点

- 业务链路完整，不只是 AI Demo
- 模型配置中心比较统一
- 资产、媒体、任务三层职责分得较清楚
- 支持全局资产复用与项目内独立演化
- 测试和守卫脚本较多，说明有回归控制意识

### 复杂度来源

- 同时支持全局资产和项目资产，两套表结构并存
- 历史 URL 与新 `MediaObject` 体系并行
- 多 provider、多模型类型、多 capability 配置交叉
- API Route 数量多，工作流路径长

### 适合理解代码的顺序

建议按下面顺序看：

1. `prisma/schema.prisma`
2. `src/lib/assets/contracts.ts`
3. `src/lib/assets/services/read-assets.ts`
4. `src/lib/assets/services/asset-actions.ts`
5. `src/lib/media/service.ts`
6. `src/lib/media/attach.ts`
7. `src/lib/config-service.ts`
8. `src/lib/generators/*`
9. `src/lib/workers/*`
10. `src/app/[locale]/workspace/*`

---

## 9. 一句话总结

当前项目的架构核心是：

“以项目和资产为业务中心，以模型配置和任务队列为 AI 执行中心，以媒体对象和统一资产摘要为数据中间层，最终把小说到视频的创作流程组织成一个可配置、可复用、可扩展的 AI 工作台。”
